# 反幻觉与质量检查清单

## 常见幻觉模式规避

| 模式 | 错误做法 | 正确做法 |
|-----|---------|---------|
| 架构虚构 | 描述原文中不存在的模型架构 | 仅引用原文Method中明确描述的架构 |
| 参数虚构 | 编造训练超参数 | 标注 `[未明确给出]` 或引用原文 |
| 结果虚构 | 声称方法能达到某精度 | 仅引用原文中报告的实验结果 |
| 仓库虚构 | 声称有开源实现 | 仅从README输入中确认的仓库 |
| 缝合虚构 | 缝合的模块在原文中不存在 | 每个缝合模块标注真实原文出处 |
| 性能承诺 | 声称方案一定有效 | 标注为 `[模型归纳·方案推演]` |

## 反幻觉检查

- [ ] 所有方法组件有原文Method来源(PMID/DOI)
- [ ] 所有架构描述来自原文或标注为 `[模型归纳·方案推演]`
- [ ] 所有训练参数来自原文或标注 `[未明确给出]`
- [ ] 所有性能数据来自原文实验结果
- [ ] 所有开源仓库来自README输入文件
- [ ] 没有编造任何论文中不存在的实现细节
- [ ] 没有虚构的benchmark结果
- [ ] **若用户有预设问题：用户的原始问题被保留为核心（未被替换）**
- [ ] **桥梁开放问题被用于补完完善用户问题（不是替代）**

## 方法缝合质量检查 ⭐

- [ ] 每个缝合模块有明确的来源方法
- [ ] 缝合点的适配方案具体可操作
- [ ] 缝合策略明确标注（组合/迁移/逆向/细化/扩展/评估创新）
- [ ] 缝合方案的创新点有清晰论证
- [ ] 方案的技术栈兼容性合理
- [ ] 无"大杂烩"式的无效缝合（无逻辑关联的模块堆砌）
- [ ] **方法来源覆盖 ≥5个不同主题方向**
- [ ] **仓库/方法来源覆盖 ≥8个不同仓库或方法**
- [ ] 方案树深度≥3层，每层有多个分支
- [ ] 有交叉互作矩阵图

## 全Pipeline完整性检查

- [ ] 数据层：数据来源/规模/获取方式/划分策略明确
- [ ] 特征层：至少2种可选方案，标注适用场景
- [ ] 模型层：核心架构Mermaid schematic + 2-3个变体Fork
- [ ] 训练层：Loss/Optimizer/Scheduler/Regularization有具体选择
- [ ] 评估层：Baselines有选择理由/Metrics有定义/Protocol有步骤
- [ ] 风险层：每个高风险有B计划
- [ ] 缝合溯源表：每个组件可追溯到原文

## 可实现性检查

- [ ] 方案所依赖的技术栈（PyTorch/TF/JAX等）标注明确
- [ ] GPU/内存需求估算合理
- [ ] 开源仓库信息从输入文件中提取，非虚构
- [ ] 无开源实现的方法标注了可复现性评估
- [ ] 数据获取路径写实不空泛

## 文献可追溯性

- [ ] 每个方法引用有DOI/PMID标识
- [ ] 方案缝合溯源表完整
- [ ] 引用格式正确
